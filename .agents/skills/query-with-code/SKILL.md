# query-with-code 技能

## 核心目标
把“当前知识库里的论文理解”和“对应代码仓库里的实际实现”放到同一条分析链里，输出一份既能解释方法、又能指导后续写代码的对照式回答。

这个技能默认做三层分析：
1. **论文层**：作者想解决什么问题、方法核心是什么、实验结论与局限是什么。
2. **代码层**：这些想法在仓库里如何落地，主干模块、训练流程、推理流程、损失函数、数据流分别在哪里。
3. **缝合建议层**：如果用户想做方法融合，判断它最适合和哪篇论文 / 哪类方法缝合，缝合点落在哪些模块，预期收益与风险是什么。

---

## 何时触发
- 用户输入 `/query-with-code`
- 用户要求“对照论文和代码仓库解释实现”
- 用户要求“找论文里的模块 A/B/C 在代码哪里”
- 用户要求“分析这篇论文在代码里是怎么实现的”
- 用户要求“帮我看模型结构、loss、训练流程、推理流程、dataset、eval”
- 用户要求“判断论文和代码哪里不一致”
- 用户要求“把论文理解转成可写代码的实现建议 / 复现路径 / 重构建议”
- 用户要求“这篇论文适合和哪篇论文 / 哪种方法融合、缝合、嫁接、组合”
- 用户要求“推荐一个最小可行缝合方案”

如果用户只给了论文、没有给代码仓库地址，或只给了代码仓库、没有说明对应论文，先停下来要求补齐最小输入。

## 用户最少输入要求

### 必需
- **论文**：标题、PDF 路径、或当前知识库里可定位的论文条目之一
- **代码地址**：GitHub 仓库链接或本地代码路径

### 强烈建议补充
- 想重点看的层面（如：模型结构 / loss / 数据预处理 / 训练 / 推理 / 评估）
- 已怀疑的关键文件
- 最想回答的问题（例如“模块 A 在哪”“训练入口在哪”“论文和代码哪里不一致”）
- 如果要做缝合分析，补一句**优化目标**（如：精度 / 效率 / 长时依赖 / 开放集泛化 / 轻量化 / 多模态 / 训练稳定性）

### 缺信息时的追问模板
优先只问最小必要问题，例如：

> 我可以做这次论文-代码对照分析，但还缺两个输入：
> 1. 要 query 的论文（标题、PDF 路径或知识库条目）
> 2. 代码仓库地址（GitHub 链接或本地路径）
> 如果你还愿意补一句“重点看模型 / loss / 训练 / 推理 / 论文与代码差异中的哪几项”，或补一句“我想优化精度 / 效率 / 泛化中的哪一项”，结果会更聚焦。

## 工作流

### 步骤 1：先查本地知识库，不能跳过
**永远先读 `wiki/index.md`。**

检索顺序：
1. 先用导航层（快速入口、按主题浏览）判断论文属于哪个主题网络
2. 再用完整注册表定位相关 `source / entity / concept / synthesis`
3. 只深读最相关的 2-4 个页面，不要一口气铺读整个库

优先读取：
- 论文对应的 `source` 页面
- 论文核心方法对应的 `entity` 页面
- 上位任务或机制对应的 `concept` 页面
- 若已有相关 `synthesis`，优先补读它来缩短分析路径

如果本地知识库找不到该论文相关内容，必须明确说：

> 本地知识库中未找到该论文的现成条目，以下将以你提供的论文信息和代码仓库为主进行分析。

如果论文已经在知识库中，回答中必须用 `[[页面名称]]` 双链引用本地页面。

### 步骤 2：确定代码源类型

#### A. 如果是本地代码路径
- 先定位入口文件、配置文件、模型目录、数据目录、训练脚本、评估脚本
- 再沿调用链回答用户问题

#### B. 如果是 GitHub / 远程仓库
- 先读取 README、主要目录、训练/推理入口说明
- 再定位与论文模块最接近的实现文件
- 必要时可补查外部文档或框架 API，但不要被框架细节带偏

### 步骤 3：把论文层和代码层强行对齐
分析时必须显式做“论文概念 → 代码实现”的映射，而不是论文讲一段、代码讲一段互不相连。

最低应回答这些问题中的大部分：
- 论文的核心模块在代码里对应哪些文件 / 类 / 函数？
- 程序从哪里启动？
- 数据是怎么进入模型的？
- forward 主路径怎么走？
- loss 在哪里计算？
- 推理 / 评估指标在哪里？
- 哪些部分明显属于论文核心，哪些更像工程包装或训练脚手架？
- 论文与代码之间是否存在“实现变体 / 简化 / 魔改 / 额外技巧”？

### 步骤 4：如果用户要求融合 / 缝合，再做方法匹配
如果用户明确询问“适合和什么论文缝合”“推荐融合对象”“该和哪条路线组合”，必须在本地知识库中先做相邻方法匹配，再落到代码实现层判断。

缝合判断顺序：
1. **先从本地知识库找候选对象**，优先同任务、同设定、相邻机制的 `entity / concept / synthesis`
2. 判断 **任务设定是否兼容**：无监督 / 弱监督 / 训练自由 / 多模态 是否能直接兼容
3. 判断 **模块是否互补**：是补 backbone、时序建模、loss、数据增强、异常评分、记忆模块，还是补事件切分 / 多模态入口
4. 判断 **代码落点是否清楚**：能替换 backbone、加 branch、加 loss、加 score fusion，还是要重写训练流程
5. 判断 **改造成本**：低成本 / 中成本 / 高成本

如果做缝合建议，至少回答：
- 推荐和哪 2-4 篇论文 / 方法融合
- 每个候选对象的互补点是什么
- 缝合点最适合落在哪些模块
- 预期收益是什么
- 主要风险是什么
- 哪个是最小可行缝合方案（MVP）

### 步骤 5：收敛成面向实现的结论
如果用户最终目标偏“写代码”而不是“读论文”，最后必须把分析转换成实现友好的总结，例如：
- 模块拆分建议
- 数据流梳理
- 伪代码层级总结
- 重构建议
- 最小可复现版本的构成

不要停留在“我帮你看懂了”，要尽量推进到“你接下来能写”。

## 推荐输出结构

优先使用这个结构回答：

```markdown
## 论文层
- 研究问题
- 核心方法
- 关键证据 / 结论
- 局限

## 代码层
- 主入口
- 模型结构
- 训练流程
- 推理 / 评估流程
- loss / dataset / 配置位置

## 论文-代码对照表
- 论文模块 A → 路径 / 类 / 函数
- 论文模块 B → 路径 / 类 / 函数

## 一致与差异
- 论文核心是否如实落地
- 哪些是工程包装 / 复现技巧 / 仓库特有改动

## 可缝合对象（如果用户要求）
- 推荐对象 1：论文 / 方法名
- 推荐对象 2：论文 / 方法名

## 缝合理由（如果用户要求）
- 互补点
- 模块落点
- 改造成本
- 预期收益
- 主要风险

## 如果要自己重写
- 建议先拆哪些模块
- 最小可复现版本包含什么
```

如果用户只关心单一层面（例如只看 loss 或只看训练入口），可以裁剪输出，但仍然要先做知识库对照。

## 强制约束
- **禁止跳过知识库检索**：必须先读 `wiki/index.md`
- **禁止只讲代码不讲论文**：这是论文-代码双层分析技能
- **禁止只讲论文不落到实现**：必须回答代码里如何实现
- **禁止空泛推荐缝合对象**：如果给出融合建议，必须说明互补点、代码落点、收益与风险
- **禁止大范围盲读整个仓库**：先定位入口与关键模块，再深读相关文件
- **禁止凭模型记忆猜论文内容**：优先依赖本地知识库、论文材料、仓库代码
- **禁止把框架胶水误判成论文创新**：必须区分论文核心与工程包装

## 最佳提问方式示例

### 示例 1：标准方式
> 帮我分析这篇论文和对应代码。  
> 论文：<标题或 PDF>  
> 代码：<GitHub 链接或本地路径>  
> 请重点看：  
> 1. 主入口  
> 2. 模型结构  
> 3. loss  
> 4. 训练 / 推理流程  
> 5. 论文和代码不一致的地方  
> 最后给我一份“如果我要自己重写，应该怎么拆模块”的总结。

### 示例 2：聚焦模块定位
> 帮我找这篇论文在代码里的主实现路径。  
> 论文：BiSP  
> 代码：https://github.com/xxx/yyy  
> 我重点想知道：forward、loss、dataset、eval 分别在哪。

### 示例 3：聚焦一致性检查
> 对照论文公式检查这个仓库有没有魔改。  
> 论文：<PDF 路径>  
> 代码：/path/to/repo  
> 重点看模型结构和损失实现。

### 示例 4：聚焦缝合建议
> 帮我分析这篇论文和对应代码，并推荐最适合缝合的论文 / 方法。  
> 论文：BiSP  
> 代码：https://github.com/xxx/yyy  
> 目标：提升开放集泛化，同时尽量少改训练流程。  
> 请输出：  
> 1. 论文层总结  
> 2. 代码层实现路径  
> 3. 适合缝合的对象（按优先级）  
> 4. 每个缝合对象的模块落点、收益、风险  
> 5. 最小可行缝合方案。

## 高价值结果固化
如果回答超过 2 个段落，且具有明显对照、总结、实现指导价值，主动询问用户：

> 这是一个适合沉淀的“论文-代码对照总结”，是否需要我把它保存为 `wiki/syntheses/` 下的实现指导页？

若用户同意，保存为 synthesis，文件名建议使用 kebab-case，例如：
- `bisp-code-reading-guide`
- `paper-code-gap-analysis-bisp`
- `vadmamba-implementation-guide`

并同步：
- 更新 `wiki/index.md` 的 `### Syntheses`
- 在 `wiki/log.md` 中追加 query 记录
