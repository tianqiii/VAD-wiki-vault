# ingest 技能

## 核心工作流：Inbox & Archive

你正在维护一个 **LLM Wiki**（Obsidian 知识库）。`raw/` 目录是"待处理收件箱"，`wiki/` 是"编译输出层"。

**目录结构约定：**
- `raw/01-articles/` — 网页剪藏的 Markdown 文章
- `raw/02-papers/` — 论文和 PDF 文献
- `raw/09-archive/` — **已处理文件的归档目录，禁止读取**
- `wiki/sources/` — 资料摘要
- `wiki/entities/` — 实体（人物、公司、工具、产品）
- `wiki/concepts/` — 概念（框架、方法论、理论）

## 触发逻辑

1. **用户执行 `/ingest`**：扫描 `raw/` 所有子目录（排除 `09-archive/`），找出待处理文件。
2. **用户执行 `/ingest <path>`**：仅处理指定文件。
3. **隐式触发**：用户说"把这个资料摄入知识库"、"导入这篇文章"时，自动执行 ingest。

## 编译流水线

对每个待处理源文件，严格按以下步骤执行：

### 预备步骤：先走确定性路径解析（推荐）

在真正开始 ingest 之前，优先使用本仓库内置脚本：

```bash
python ".agents/scripts/router.py" ingest [可选参数]
```

读取其 JSON 输出，优先使用里面给出的：
- `workspace_root`
- `wiki_dir`
- `raw_dir`
- `index_path`
- `log_path`

作用：
- 避免在不同工作目录下靠模型猜测路径
- 在真正 ingest 前先确认 `wiki/`、`raw/`、`index.md`、`log.md` 是否存在
- 让后续步骤直接围绕稳定路径执行

### 0) 先判定资料类型（论文优先）

- `raw/02-papers/*.pdf` → **论文模式（Paper Mode）**：遵循本仓库当前的极简模板（稳定的 frontmatter + 固定章节），但在内容上按论文习惯抽取“问题-方法-证据-局限”。
- `raw/01-articles/*.md` 等 → 通用模式：按现有流程提炼并建链。

### 步骤 1：读取源文件

- **如果是 `.md` 文件**：使用读取工具完整读取内容。
- **如果是 `.pdf` 文件**：使用读取工具尝试提取文本。
  - 若提取失败/内容为空：仍要创建 `wiki/sources/` 摘要页，但必须在正文显式标注 `需要人工补读`（只写文件名、页数、你能读到的可见元信息），并**禁止归档**该 PDF（避免把“未真正摄取”的文件移动走）。

### 步骤 2：提炼核心并翻译

从源文件中提取：
- **核心动机**：论文的主要想解决什么问题
- **创新点**：为解决问题提出的创新性方法
- **实体**：人物、公司、工具、产品等具体名词
- **概念**：框架、方法论、理论等抽象名词

如果是非中文内容，则翻译成中文。

#### 论文模式（raw/02-papers）必抽取清单（写进正文与关键信息，不强制塞进 frontmatter）

从论文文本中，至少抽出这些点（缺失要明确写“未在正文中找到/需补读”）：

- **研究问题**：要解决的异常/任务定义是什么？
- **核心方法**：用什么机制做检测？关键模块/假设是什么？
- **证据与评估**：数据集、指标、对比基线、主要结果（只保留最关键 2-3 条）。
- **局限与适用边界**：作者承认的局限 + 你判断的适用条件。
- **可复现性线索（若能读到）**：是否有代码/数据/权重链接。

### 步骤 3：创建来源摘要

在 `wiki/sources/` 创建 Markdown 文件：

```markdown
---
title: "摘要-文件slug"
type: source
tags: [来源, 原始文件]
sources: ["raw/01-articles/xxx.md"]
last_updated: YYYY-MM-DD
---

## 核心摘要
[3-5句话的核心总结]

## 关联连接
- [[EntityName]] — 关联实体
- [[ConceptName]] — 关联概念
```

文件名使用 kebab-case：`摘要-{文件slug}.md`

#### 论文 source 页（强制贴合本仓库现有模板）

当源文件来自 `raw/02-papers/*.pdf` 时，source 页必须使用如下模板（这是当前仓库已存在的事实标准）：

```markdown
---
title: "摘要-<paper-slug>"
type: source
tags: [来源, 论文, <主题标签>]
sources: ["raw/09-archive/<论文PDF文件名>.pdf"]
last_updated: YYYY-MM-DD
---

## Metadata
- **作者**: 作者1, 作者2, ...
- **年份**: YYYY
- **期刊/会议**: 期刊名或会议名
- **DOI / arXiv**: 有则写，无则标注“未在正文中找到”

## 核心摘要
（一段高密度总结：问题→方法→证据/结果→局限/适用边界）

## 关联连接
- [[MethodEntity]] — 论文对应的核心方法实体（若有）。
- [[TaskOrConcept]] — 上位任务或可复用概念。
```

注意：
- `sources` 必须是数组（即使只有一个 PDF）。
- **默认必须写 `## Metadata` 区块**，优先从论文首页、页眉、bib 信息或 PDF 可见元信息提取作者、年份、期刊/会议、DOI / arXiv。
- 不要把这些元信息塞进 frontmatter；继续保持 frontmatter 极简，把论文 metadata 放在正文的 `## Metadata` 中。
- 若某项缺失，不要空着，明确写“未在正文中找到”或“需人工补读”。

#### Paper Mode 元信息提取要求（新增）

对于 `raw/02-papers/*.pdf`，除“问题-方法-证据-局限”外，还必须尽量提取这些 metadata：

- **作者**：按论文显示顺序记录
- **年份**：优先取论文正式发表年份；若只能确认 arXiv 年份，也要明确写出
- **期刊/会议**：例如 CVPR、ICCV、NeurIPS、KDD、TNNLS、PR 等
- **DOI / arXiv**：有则记录；没有则明确标注未找到

提取原则：
- 优先准确，不要为了“看起来完整”而猜测
- 如果只能读到部分元信息，也照实写出，不要静默省略
- 如果论文是 survey / tutorial / workshop / arXiv preprint，也照原始可见元信息写明

### 步骤 4：知识网络化（实体/概念页面）

对于步骤 2 提取的每个实体和概念：

**目标目录：**
- 实体 → `wiki/entities/`
- 概念 → `wiki/concepts/`

**处理逻辑：**
1. 页面不存在 → 按照 AGENTS.md 的 Frontmatter 规范创建新页面
2. 页面已存在 → 读取现有内容，**增量合并**新信息
3. **发现冲突** → **立即暂停**，向用户报告冲突内容，询问处理方式后再继续

**页面模板：**

```markdown
---
title: "页面名称"
type: entity | concept
tags: [标签]
sources: ["raw/09-archive/xxx.pdf"]
last_updated: YYYY-MM-DD
---

## 定义
[对该实体/概念的定义]

## 关键信息
[从源文件中提取的详细信息]

## 关联连接
- [[摘要-source-slug]] — 来源
- [[RelatedEntity]] — 相关实体
```

#### 论文模式下的实体/概念建模规则（避免页面爆炸）

- **方法实体（entity）**：
  - 若论文提出明确命名的方法/系统（例如某个 VAD 方法名）：建立 1 个核心方法实体页（TitleCase）。
  - 若是综述/调查（survey）：一般不强制创建实体页，优先把“任务概念/子方向概念”补齐。
- **概念（concept）**：
  - 优先复用已存在概念（先全库检索相近标题再新建）。
  - 只把“可跨论文复用”的东西建成概念页（如 `StateSpaceModel`、`PseudoAnomalyGeneration`），不要把每篇论文的一次性短语都新建成概念。
- **双向链接**：
  - source → 至少链接 1 个方法实体（如有）+ 1 个上位概念。
  - entity/concept → 必须反向链接回该 source（避免孤岛页）。

### 步骤 5：更新全局索引（完整注册表必更，导航层按需补录）

**更新 `wiki/index.md`：**
`wiki/index.md` 采用两层结构，更新规则必须区分：

1. **完整注册表（必须更新）**
   - 所有本次新增的 source / entity / concept / synthesis 页面，都必须登记到 `## 完整注册表` 下对应区块。
   - 注册条目格式：`[[页面名称]] — 一句话描述`
   - 示例：
     - Sources: `[[摘要-source-slug]] — 该资料的核心主旨`
     - Entities: `[[EntityName]] — 该实体的身份定义`
     - Concepts: `[[ConceptName]] — 该概念的核心定义`

2. **导航层（按高复用价值补录，不是每页都进）**
   - 只有当页面具备明显的“入口价值”时，才补入 `快速入口` 或 `按主题浏览`。
   - 典型应补入导航层的页面：
     - 能充当主题枢纽的上位概念（如任务总入口、监督设定、核心机制）
     - 能代表某条研究路线的关键方法实体
     - 新增后会显著改善用户/Agent 的检索收敛路径的页面
   - 通常**不需要**补入导航层的页面：
     - 仅服务于单篇论文的 source 页
     - 一次性、低复用、只在局部成立的概念
     - 与现有导航层高度重复、不会显著改善入口能力的页面

#### 导航层补录判断（推荐启发式）

如果新增页面满足以下任意一条，可考虑补入导航层：
- 它是某个主题网络的总入口或上位概念
- 它已经被多个页面链接，且后续大概率会继续被复用
- 用户后续很可能会直接围绕它提问（适合作为 query 第一跳）
- 缺少它会让当前主题只能通过阅读长注册表才能找到

如果拿不准：
- **先只更新完整注册表**，不要为了“看起来完整”把大量页面塞进导航层
- 导航层追求的是“少而准的入口”，不是“覆盖全部页面”

**更新 `wiki/log.md`：**
追加操作日志（Append-only）：
```markdown
## [YYYY-MM-DD] ingest | 操作简述
- **变更**: 新增 [[PageName]]; 更新 [[index.md]]
- **冲突**: 无 (或: 冲突 [[ConflictingPage]], 已暂停等待决策)
```

### 步骤 6：归档源文件

在确认以下全部完成后，将源文件移动到 `raw/09-archive/`目录：
- sources 页面已创建
- 实体/概念页面已创建或更新
- index.md 已更新
- log.md 已更新

**绝对禁止修改源文件内部的文字。**

#### 归档一致性要求（必须做，避免“路径漂移”）

归档完成后，必须保证：本次 ingest 触及的所有 wiki 页面（source/entity/concept）其 frontmatter `sources:` 字段都指向 **归档后的路径** `raw/09-archive/...`，而不是 `raw/02-papers/...`。

推荐做法：
- 在创建/更新 wiki 页时就直接写 `raw/09-archive/<文件名>`（因为你知道归档目标目录与文件名）。
- 如果先写了 `raw/02-papers/...`（不推荐），则在移动文件后，必须立刻把本次触及的页面里的 `sources:` 一并更新到 `raw/09-archive/...`。

最低校验：
- 对 `wiki/**/*.md` 搜索 `raw/02-papers/`，结果必须为 0。

## 冲突处理流程

当发现新旧知识冲突时：

1. **暂停**：停止当前 ingest 流程
2. **报告**：向用户说明冲突内容（哪个页面、冲突点是什么）
3. **询问**：请用户选择处理方式：
   - A) 保留新旧两者，标注为"知识冲突"
   - B) 用新知识覆盖旧知识
   - C) 放弃本次 ingest
4. **继续**：根据用户选择继续或终止

## 注意事项

- 绝对不读取 `raw/09-archive/` 下的任何文件
- 所有 wiki 页面必须包含 `## 关联连接` 区域，不能产生孤岛页面
- 使用简体中文编写所有内容
- 实体命名使用 TitleCase，概念和来源使用 kebab-case

## 快速自检（论文模式强烈建议执行）

- [ ] `wiki/sources/摘要-*.md` 是否符合现有模板（frontmatter 五字段 + 两个二级标题）
- [ ] 新增/更新的 entity/concept 是否都有 `定义/关键信息/关联连接`
- [ ] `wiki/index.md` 的完整注册表是否登记了新增页
- [ ] 若页面具备高复用价值，是否已判断要不要补入导航层
- [ ] `wiki/log.md` 是否 append 了本次 ingest
- [ ] `wiki/` 中是否不存在 `raw/02-papers/` 的 sources 引用
- [ ] 若 PDF 提取失败：是否标注“需要人工补读”，并且没有归档
